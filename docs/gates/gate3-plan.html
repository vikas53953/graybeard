Graybeard — architecture & build plan

Your R9 law applies to us first: here is Graybeard's own System Design — the map and the principles — then the build plan in six work units. Decisions you might want to change are at the top; plumbing I decided by principle is at the bottom, logged.

System Design — the map (what Graybeard is made of)

🖥️ ScreensHTML pages like this one — gates, evidence, dashboards. No app UI to install.
🧠 BrainThe skills — rulebooks written in plain text the AI must follow. This is 90% of the product.
🗄️ MemoryOne profile file per user (their answers, their laws) + one scoreboard per project.
🔑 KeysSKIPPED — no accounts, no login. It's files on your machine.
🏠 HomeGitHub repo + the Claude plugin marketplace. Two commands to install.
🔭 WatchtowerEvery law carries its test evidence; every deviation goes in a ledger. The system can explain itself.
🔒 LocksTranscript reading is 100% local · every touched file backed up first · exact uninstall via a manifest.
🔌 PipesOnly Claude Code's own plugin surface (skills, hooks, CLAUDE.md). No outside services.

System Design — the principles (why it's built this way)

✂️ One job per file — each skill does one thing; no god-files (the Touch-the-Grass killer). · 📍 One source of truth — the user's answers live in ONE profile file; everything else is generated from it. · 🎯 Simplest thing that works — the whole product is text files; zero code to compile, zero servers. · ➡️ One-direction flow — templates → generated personal files → behavior; never backwards. · 🔒 Security at the boundary — the only risky touchpoint (the user's settings file) gets backup + merge + verified restore. · 🔭 Observability — laws ship with proof, changes ship with ledger entries. · 🧩 Loose coupling — plugin updates can never overwrite personal laws. · 🗑️ Design-for-deletion — a manifest records every generated file; uninstall removes exactly those.
